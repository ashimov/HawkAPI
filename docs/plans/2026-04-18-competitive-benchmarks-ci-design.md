# Competitive benchmarks CI + publishing — design spec

**Status:** Approved — ready for implementation
**Date:** 2026-04-18
**Scope:** Wire the existing `benchmarks/competitive/` suite into GitHub Actions, automate `RESULTS.md` regeneration, and surface the numbers in user-visible docs.

**Three-expert deliberation output (earlier in session):** "Linux benchmarks" was flagged as the second priority after DX gaps because visible numbers turn technical wins into market wins.

---

## Goal

After this PR lands:

- Every release + every Monday 06:00 UTC, a GitHub Actions job runs the full `./benchmarks/competitive/run_all.sh` on an `ubuntu-latest` runner.
- `benchmarks/competitive/RESULTS.md` is regenerated and committed back to main by `github-actions[bot]` via PR (not direct push).
- Manual trigger available via `workflow_dispatch`.
- `README.md` shows a visible "Performance" section with the current top-line numbers.
- `docs/guide/benchmarks.md` documents methodology, current numbers, and reproduction steps.
- Release artefacts include `RESULTS.md` + raw JSON.

## Existing infrastructure (already in the repo)

- `benchmarks/competitive/apps/` — one app file per framework (HawkAPI, FastAPI, Litestar, BlackSheep, Starlette, Sanic).
- `benchmarks/competitive/runner.py` — starts Granian, pounds endpoints with `wrk`, collects RPS + p50/p99 latency.
- `benchmarks/competitive/run_all.sh` — orchestrates all (framework × scenario) combinations.
- `benchmarks/competitive/generate_report.py` — aggregates results into `RESULTS.md`.
- `benchmarks/competitive/RESULTS.md` — current snapshot (2026-04-17). HawkAPI wins 5/6 scenarios by throughput.
- `benchmarks/competitive/README.md` — methodology + reproduction docs.
- 6 scenarios: `json`, `plaintext`, `path_param`, `body_validation`, `query_params`, `routing_stress`.
- `.github/workflows/benchmark.yml` — **outdated**: runs the micro-benchmark scripts (`bench_request_response.py`, `e2e/run_benchmarks.py`) on release only; never the competitive suite.

## What changes

### 1. `.github/workflows/benchmark.yml` — overhaul

Replace the body:

```yaml
name: Competitive benchmarks

on:
  release:
    types: [published]
  schedule:
    # Every Monday at 06:00 UTC — keeps RESULTS.md within one week old.
    - cron: "0 6 * * 1"
  workflow_dispatch:
    inputs:
      duration:
        description: "wrk duration per scenario (seconds)"
        required: false
        default: "10"
      connections:
        description: "wrk concurrent connections"
        required: false
        default: "64"

permissions:
  contents: write       # for auto-commit PR
  pull-requests: write  # for peter-evans/create-pull-request

jobs:
  competitive:
    name: Run competitive suite + publish RESULTS.md
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install wrk
        run: |
          sudo apt-get update
          sudo apt-get install -y wrk

      - name: Install uv
        uses: astral-sh/setup-uv@v4
        with:
          enable-cache: true

      - name: Set up Python 3.13
        run: uv python install 3.13

      - name: Install framework deps
        run: uv sync --extra dev --extra bench --extra uvicorn --extra granian --extra pydantic

      - name: Run competitive benchmark suite
        env:
          DURATION: ${{ inputs.duration || '10' }}
          CONNECTIONS: ${{ inputs.connections || '64' }}
        run: ./benchmarks/competitive/run_all.sh

      - name: Upload raw results + RESULTS.md
        uses: actions/upload-artifact@v4
        with:
          name: competitive-results
          path: |
            benchmarks/competitive/results/
            benchmarks/competitive/RESULTS.md
          retention-days: 90

      - name: Open PR with refreshed RESULTS.md
        if: github.event_name != 'release'
        uses: peter-evans/create-pull-request@v7
        with:
          add-paths: benchmarks/competitive/RESULTS.md
          branch: chore/refresh-benchmarks
          commit-message: "chore(bench): refresh competitive RESULTS.md"
          title: "chore(bench): refresh competitive RESULTS.md"
          body: |
            Auto-generated by the `benchmark` workflow (${{ github.event_name }} event).

            wrk config: ${{ inputs.duration || '10' }}s × ${{ inputs.connections || '64' }} connections.
          delete-branch: true

      - name: Attach RESULTS.md to GitHub release
        if: github.event_name == 'release'
        uses: softprops/action-gh-release@v2
        with:
          files: benchmarks/competitive/RESULTS.md
```

Design decisions:

- **PR rather than direct commit** — review loop prevents accidental regressions sneaking in as "benchmark moved, looks fine."
- **Release event attaches to release assets** (not PR) — releases are immutable and the numbers at release time are historical.
- **Weekly schedule** — catches performance drift without being spammy.
- **Manual dispatch with custom duration/connections** — useful for ad-hoc deep runs.

## Out of scope

- **Charts / graphs** — RESULTS.md table is sufficient for v1; charting adds complexity.
- **Cross-OS matrix** — ubuntu-latest is "Linux benchmarks" by definition.
- **Memory benchmarks in competitive suite** — separate `bench_memory.py` already exists.
- **Dockerised benchmarks** — follow-up.
- **Historical trend graphs** — would need a results DB; not v1.
- **Adding more frameworks** (Quart, Emmett, Django+asgiref) — deferrable.
- **Tuning app implementations** — apps exist; sanity-check only, no optimisation work here.

## Success criteria

1. `workflow_dispatch` of "Competitive benchmarks" on main runs to completion on `ubuntu-latest`.
2. RESULTS.md is regenerated and a PR is opened against main with the refreshed content.
3. A GitHub release triggers the workflow and attaches RESULTS.md as an asset.
4. README's Performance section renders correctly on github.com.
5. `docs/guide/benchmarks.md` appears in the Guide nav; mkdocs strict build clean.
6. No regressions in the normal CI suite.
